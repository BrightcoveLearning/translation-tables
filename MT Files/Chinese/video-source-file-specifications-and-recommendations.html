    視訊來源檔規格與建議

    Brightcove 建議您對上傳至媒體庫的視訊檔案使用下列編碼規格。

    簡介

    Brightcove 建議您對上傳至媒體庫的視訊檔案使用下列編碼規格。

    Video Cloud支持H.264和H.265（請參閱混合動力汽車下面）的編碼。一般而言，建議使用 H.264，因為它提供更廣泛的裝置支援。H.264 視訊編碼將使用 AAC 音訊。

    如果您不熟悉Video Cloud的媒體模塊，請閱讀媒體模塊入門，其中說明了媒體模塊的基本功能以及如何啟動它。另請閱讀使用上傳模塊上傳視頻，以獲取有關如何將視頻上傳到媒體模塊的說明。

    看到支持的視頻格式，編解碼器和容器有關Brightcove的視頻格式的信息不支持（數量不多）。

    Brightcove 建議您對上傳至媒體庫的視訊檔案使用下列規格：

    • 源文件命名
    • H.264（MPEG-4 Part 10）編碼建議
    • 高分辨率編碼建議
    • 移動應用程序編碼
    • 保持寬高比

    HEVC (H.265) 影片

    高效率視訊編碼 ( HEVC ) 也稱為 H.265，是視訊壓縮標準，設計成廣泛使用的進階視訊編碼 (AVC/H.264) 的後續廠商。與 AVC 相比，HEVC 在相同等級的視訊品質下，提供大約 20% 的資料壓縮，或以相同位元速率大幅改善視訊品質。

    這項改良壓縮的優點包括降低頻寬使用量、以較低的位元速率提供更高的視訊品質，以及縮短第一個影格 (TTFF) 的時間，這一切都會提高終端檢視者滿意度。

    缺點

    主要缺點是，因為它是一種新技術，HEVC 視頻尚未被客戶端廣泛支持（意味著設備 + OS + 瀏覽器/應用程序）。此外，有些用戶端可以在播放期間切換 HEVC 和 AVC 區段，而其他用戶端則無法切換。

    此外，HEVC 影片的轉碼時間比 H.264 還要花費大約 2-4 倍的時間。

    可用性

    HEVC支持是付費附件視頻雲。如果您對 HEVC 支援有興趣，請聯絡您的客戶經理。

    來源檔案命名

    在擷取程序期間，必須將來源檔案名稱作為 API 要求 URL 的一部分傳遞。因此，名稱必須與 URI 相容。請避免在所有視訊和資產的來源檔案名稱中使用下列項目：

    • 空格：而不是my video.mov，使用my-video.mov或my_video.mov
    • 特殊字元：請勿在來源檔案名稱中使用下列字元："!", "@", "#", "$", "%", "^", "&", "*", "(", ")", "?", "/", "\", "{", "}", "[", "]"

    H.264 編碼建議

    視頻編解碼器： h264
    編碼方式： 兩次通過 H.264
    關鍵幀： 至少每 6 秒

    高解析度編碼建議

    若要進行高解析度的影片轉譯，請上傳 H.264 來源檔案至視訊雲端。布萊特灣測試了不同大小和寬高比的 H.264 檔案。以下每種情況都假設使用 16 位元 AAC 音訊的 H.264 2 通道編碼，採樣頻率為 44.1kHz，而位元速率為 192 kbps。所有位元速率不包含音訊位元速率，只包含視訊。在討論中，我們會補償音訊，並提供視訊上傳至伺服器端轉碼服務的近似最大長度。如果視訊交錯，視訊雲端會在轉碼期間解除交錯。如果您尚未使用 Video Cloud 進行影片轉碼，請先解除交錯的影片檔案，再上傳影片檔案。關鍵畫面的最大距離應為 6 秒，且不需要等距。

    大小 長寬比 建議的最小位元速率
    640x480 4:3 動作少許的視訊 (例如說話頭) 可以低至 900 kbps，效果非常好。無論動作程度如何，都能獲得整體高品質，請使用至少 2672 kbps。這也指的是 720x480 源，因為它們的意思是 640x480，但具有不同的像素寬高比。
    640x360 16:9 在幾乎所有情況下，視訊位元速率都能安全地降低至 2132 kbps。
    1024x768 4:3 此格式主要用於螢幕廣播或裁切的 HD 來源。在視頻的情況下，4864 kbps 就足夠了。在屏幕截圖的情況下，我們並不需要那麼多的數據，因為從幀到幀的變化很少。我們可以擺脫 720p 的低端，這是 3136 kbps。這實際上比我們需要的要多一些，但我們建議使用更高的位元速率，以允許更頻繁的關鍵影格。因為我們不想要鼠標的奇怪鬼影，所以最好有非常頻繁的關鍵幀放置等距，每秒兩次（所以在非常平滑的屏幕截圖的情況下，每 15 幀）。如果屏幕截圖的幀速率不是 30（或 29.97），則可以平行調整數據速率。所以，如果你有一個 15 幀速率的屏幕轉播，數據速率將是 1568 kbps。這對於螢幕廣播來說是很大的頻寬，但是有必要消除移動到 H.264 之類的東西時常發生的鬼影和頁面撕裂。
    1280x720 16:9 這種格式的數據速率可以低至 3136 kbps，對於一個說話的頭。當運動增加時，即使是一點點，比特速率必須增加相當多的補償。在低動作的情況下（意味著主體移動，但相機/框架沒有），假設照明充足或理想，4512 kbps 就足夠了。對於高動作或補償其他因素，5928 kbps 就足夠了（有一點呼吸空間）。
    1920x1080 16:9 拉伸來源 許多 HDV 和 AVCHD 相機，即使是專業級別，也使用 4:3 感應器，並將影像從 1440x1080 延伸至 1920x1080，以便偽裝高畫質。最後，這通常會產生在 960x540 下幾乎同樣清晰的影像。考慮到這一點，這種格式將在 960x540 和 1920x1080 中提供建議。為了妥協，請在 720p 中上傳，先前的 720p 設定適用於此處。該格式需要 8024 公里的位元速率來處理高動作。說話的頭可以安全地降低至 6000 kbps，但 8024 kbps 應該涵蓋大多數情況。這將允許一點點超過半小時。使用 960x540 格式，如果您沒有升頻到更大的尺寸，並且運動不太重，您可以使用與 720p 說話頭相同的 3136 kbps 速率。

    行動應用程式的編碼

    H.264 是目前行動應用程式的最佳編碼標準。行動裝置支援 H.264 的基準設定檔。除了設備的實際視頻播放功能之外，還考慮網絡條件和它們對下載速度的影響。我們建議您至少在 130 kbps 以下播放一個節目，以便在行動網路上進行可靠的播放。這對於大多數用戶來說應該可靠地執行，同時保持盡可能多的質量。

    維護外觀比例

    Video Cloud 將影片轉碼時，它會維持與來源檔案相同的外觀比例。對於相反的來源檔案 (有時稱為具有非方形像素)，Video Cloud 會偵測並修正這個問題，讓輸出視訊具有正方形像素，以確保在桌上型電腦和行動裝置上正確播放。

    音訊

    對於音訊，我們建議使用 AAC 格式。此外，音軌的持續時間應該與視頻軌道相同-如果它們不匹配，視頻仍然會被攝入，但在某些情況下，您可能會看到播放失敗。

    音軌

    僅支持一個音軌，除非為您的帳戶啟用了多個音軌，在這種情況下，多個音軌將被視為替代音軌。否則，Video Cloud 會使用視訊檔案中的第一個音軌，並略過其他任何音軌。

    音訊頻道

    支援 1 或 2 個音訊頻道。如果有額外的音頻頻道，它們將被合併為 2-結果往往令人不滿意。
